## 1. Summary

Refactor Flux compiler architecture into explicit, testable phase modules first, then layer performance improvements on top.  
Current compiler complexity is concentrated in:
- `src/bytecode/compiler/mod.rs` (~2178 LOC)
- `src/bytecode/compiler/expression.rs` (~2625 LOC)

This proposal defines a behavior-preserving phase split (Milestone 1), then a measured performance program (Milestones 2â€“4) with hard regression gates for effect/type diagnostics parity.

---

## 2. Motivation

### Current pain points

1. **Monolithic orchestration:** `compile()` mixes state reset, inference, validations, and codegen.
2. **Implicit phase contracts:** pass order is encoded only by call order, hard to verify and evolve.
3. **Repeated full-program scans:** contracts/effects/ADTs/validations walk the program independently.
4. **Perf tuning risk:** architecture opacity makes â€śfastâ€ť changes likely to break diagnostics behavior.

### Why architecture-first

A clean pass pipeline is the prerequisite for safe and measurable performance optimization in a compiler with strict diagnostic contracts (Aâ€“G from 043).

---

## 3. Goals

1. Make compiler phases explicit and composable.
2. Preserve existing language semantics and diagnostics.
3. Establish measurable compile-time performance baseline and improvement targets.
4. Enable focused optimization work without destabilizing type/effect correctness.
5. Keep VM/JIT compile-time diagnostics parity intact.

---

## 4. Non-Goals

1. New language features or syntax changes.
2. Replacing HM inference/effect-row algorithms in this proposal.
3. Runtime/JIT execution optimization of generated bytecode.
4. Rewriting `expression.rs` fully in one pass.

---

## 5. Proposed Architecture

### 5.1 Phase Pipeline Model

Introduce a pass runner with explicit ordered phases:

1. `Prepare`  
2. `CollectContractsAndDecls`  
3. `InferUnannotatedEffects`  
4. `ValidateEntrypointAndPurity`  
5. `PredeclareFunctions`  
6. `InferTypesHM`  
7. `ValidatePatterns`  
8. `CodegenStatements`  
9. `PostprocessDiagnostics`  
10. `FinalizeEntrypointEmission`

Each phase receives and updates a typed `PipelineContext` + `PhaseArtifacts`.

### 5.2 New Internal Modules

Create:
- `src/bytecode/compiler/pipeline.rs`
- `src/bytecode/compiler/passes/mod.rs`
- `src/bytecode/compiler/passes/prepare.rs`
- `src/bytecode/compiler/passes/collect.rs`
- `src/bytecode/compiler/passes/effects_infer.rs`
- `src/bytecode/compiler/passes/validate.rs`
- `src/bytecode/compiler/passes/predeclare.rs`
- `src/bytecode/compiler/passes/hm.rs`
- `src/bytecode/compiler/passes/patterns.rs`
- `src/bytecode/compiler/passes/codegen.rs`
- `src/bytecode/compiler/passes/diag_post.rs`

`mod.rs` becomes a thin facade over pipeline execution and compiler state.

### 5.3 Phase Artifacts

Define artifacts to avoid recomputation:
- `ContractIndex`
- `EffectDeclIndex`
- `AdtIndex`
- `MainValidationState`
- `HMResult { type_env, diagnostics }`
- `PatternDiagnostics`
- `CodegenResult`

## 6. Performance Roadmap (Post-Split)

### 6.1 Baseline first

Add compile-time benchmarks and tracing:
- time each phase
- count program walks
- count cloned AST/program values
- collect diagnostic count and ordering hashes

### 6.2 Optimization phases

#### Phase P1: Remove avoidable clones
- Minimize `program.clone()` in `compile_with_opts`
- Isolate optimization/analyze paths to avoid cloning for normal compile

#### Phase P2: Consolidate scans
- Merge declaration/contract/effect/ADT collection into one indexed prepass

#### Phase P3: Hotpath caching
- Cache repeated effect inference pieces where safe (e.g., contract effect row expansion)
- Reduce transient `HashSet/Vec` churn in inference paths

#### Phase P4: Scope/symbol-table efficiency
- Reduce `SymbolTable` cloning during scope enter/leave with incremental overlays

All optimization phases must preserve diagnostics invariants.

## 7. Public/Internal Interface Changes

### 7.1 External APIs (public behavior)

No user-facing CLI/language changes.

### 7.2 Internal compiler interfaces

New:
- `CompilerPipeline`
- `PipelineContext`
- `Phase` enum + `run_phase()`
- `PhaseArtifacts`

Changed:
- `Compiler::compile()` delegates to pipeline
- existing helper methods moved into pass modules with narrower signatures

## 8. Diagnostics Compatibility Contract

Must preserve:

- primary label text
- deterministic ordering class (especially entry-policy + strict diagnostics)

- `purity_vm_jit_parity_snapshots`
- existing type/effect fixture snapshots

Any intentional diagnostic text changes require snapshot review and explicit proposal note.

## 10. Rollout Plan

### Milestone 1 (Architecture split, behavior-preserving)
- introduce pipeline + pass modules
- move logic with minimal edits
- keep all tests green

### Milestone 2 (Measurement harness)

### Milestone 3 (Low-risk perf wins)

- artifact reuse

### Milestone 4 (Targeted hotpath optimization)
- expression/effect inference memory churn reduction

## 12. Risks and Mitigations

1. **Risk:** Subtle behavior drift during extraction  
   **Mitigation:** phase-by-phase moves + snapshot parity gating.

2. **Risk:** Over-refactor delays performance gains  
   **Mitigation:** fixed Milestone 1 scope; no semantic rewrites.

3. **Risk:** Perf regressions from extra abstraction  
   **Mitigation:** baseline-first metrics and per-phase timers.

4. **Risk:** Diagnostic ordering changes  
   **Mitigation:** explicit ordering tests and parity snapshot checks.
