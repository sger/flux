# Proposal 0106: Standalone Binary Emission

## Summary

Enable `flux build program.flx -o program` to produce a standalone native executable
that runs without the Flux compiler, VM, or JIT installed. The LLVM backend compiles
Flux source to a `.o` object file (already implemented), and a Rust-based linker driver
links it against a static Flux runtime library to produce a self-contained binary.

---

## Motivation

Today Flux programs can only run through the compiler:

```bash
cargo run -- program.flx           # VM (interpreted)
cargo run --features jit -- program.flx --jit    # Cranelift JIT
cargo run --features llvm -- program.flx --llvm  # LLVM JIT
```

All three modes require the full Flux compiler binary. You can emit a `.o` object
file with `--emit-obj`, but it can't run on its own — it depends on 50+ `rt_*`
runtime symbols (`rt_add`, `rt_make_string`, `rt_perform`, etc.) that only exist
inside the compiler process.

This blocks:

1. **Distribution** — users can't ship a Flux program as a single binary
2. **Deployment** — servers/containers need the full Flux toolchain installed
3. **Performance** — AOT-compiled code with LLVM O2/O3 is faster than JIT, but
   only accessible via `--emit-obj` which produces an unusable `.o` file
4. **Interop** — embedding Flux-compiled code in Rust/C projects requires manually
   linking against undocumented runtime symbols

### Use cases

```bash
# Build and run a standalone binary
flux build examples/basics/fibonacci.flx -o fibonacci
./fibonacci

# Build with optimizations
flux build examples/basics/fibonacci.flx -o fibonacci -O

# Cross-compile (future)
flux build program.flx -o program --target aarch64-unknown-linux-gnu
```

## Guide-level explanation

### Building a standalone binary

```bash
flux build hello.flx -o hello
./hello
```

This compiles `hello.flx` through the full pipeline (parse → type inference →
Core IR → CFG → LLVM IR → machine code), links it against the Flux runtime
library, and produces a native executable.

The resulting binary:
- Contains the compiled Flux program as native machine code
- Includes the Flux runtime (GC, base functions, effect handlers, value system)
- Runs without any Flux toolchain installed
- Is a standard ELF/Mach-O/PE binary that can be stripped, profiled, debugged

### Build flags

| Flag | Description |
|------|-------------|
| `-o <path>` | Output path for the executable |
| `-O` | Enable LLVM O2 optimizations |
| `--emit-obj` | Emit `.o` only (existing behavior, no linking) |
| `--emit-asm` | Emit `.s` assembly only |
| `--static` | Statically link all dependencies (default) |

## Reference-level explanation

### Architecture

```
Source (.flx)
  → Flux compiler pipeline (parse, infer, Core IR, CFG)
  → LLVM backend (compile_program_ir_only)
  → Object file (.o) with __flux_entry symbol
  → Link with libflux_runtime.a + Rust entry shim
  → Standalone executable
```

### Phase 1: Runtime static library (`libflux_runtime.a`)

Create a new crate target that compiles the Flux runtime as a static library:

```toml
# Cargo.toml
[[lib]]
name = "flux_runtime"
crate-type = ["staticlib"]
```

The library exposes:
- All `rt_*` extern "C" functions from `runtime/native_helpers.rs`
- `flux_runtime_init() -> *mut JitContext` — create and initialize a JitContext
- `flux_runtime_destroy(*mut JitContext)` — teardown and free
- `flux_runtime_register_functions(*mut JitContext, entries: &[JitFunctionEntry])` — register compiled function table

### Phase 2: Rust entry shim

Generate a Rust `main.rs` that bootstraps the runtime and calls `__flux_entry`:

```rust
// Generated by `flux build`
extern "C" {
    fn __flux_entry(ctx: *mut std::ffi::c_void) -> [i64; 2];
}

fn main() {
    unsafe {
        let ctx = flux_runtime::init();
        flux_runtime::register_functions(ctx, include_bytes!("function_table.bin"));
        let result = __flux_entry(ctx as *mut _);
        // Handle JIT_TAG_THUNK trampoline loop
        flux_runtime::trampoline(ctx, result);
        flux_runtime::destroy(ctx);
    }
}
```

### Phase 3: Linker driver

The `flux build` command orchestrates:

1. Compile `.flx` → `.o` via LLVM backend (`compile_program_ir_only`)
2. Serialize function metadata (contracts, return spans, arity) to a side table
3. Write the generated `main.rs` to a temp directory
4. Invoke `rustc` (or `cc`) to compile `main.rs` + link `program.o` + `libflux_runtime.a`
5. Produce the final executable

```bash
# What `flux build hello.flx -o hello` does internally:
flux --llvm --emit-obj hello.flx              # → hello.o
rustc main_shim.rs hello.o -L. -lflux_runtime -o hello
```

### Phase 4: Function table serialization

The compiled `.o` file contains machine code for user functions, but the runtime
needs metadata (arity, contracts, return spans) to dispatch calls correctly. This
metadata is serialized as a binary blob and embedded in the entry shim via
`include_bytes!`.

The `JitFunctionEntry` table maps function index → { pointer, arity, contract, span }.
At startup, the entry shim calls `flux_runtime_register_functions` to populate the
`JitContext.jit_functions` vector with these entries, using symbol addresses resolved
by the linker.

### Binary size

Expected sizes for a minimal Flux program:

| Component | Size |
|-----------|------|
| Compiled Flux code | 1-50 KB |
| Flux runtime (GC, base fns, values) | ~2-3 MB |
| Rust stdlib (if statically linked) | ~3 MB |
| **Total (stripped)** | **~5 MB** |

With `--release` and `strip`: ~3-5 MB for a minimal program. Comparable to a Go binary.

---

## Drawbacks

1. **Binary size** — the Flux runtime adds ~3 MB to every binary. Unlike C where
   you only link what you use, the Flux runtime is monolithic (GC, all 77 base
   functions, effect handler machinery).

2. **Requires Rust toolchain** — linking currently needs `rustc` available. Could
   use `cc` instead, but then Rust std library linking is manual.

3. **No dynamic linking** — each binary bundles its own runtime. Multiple Flux
   binaries on the same system duplicate the runtime.

4. **Debug info** — mapping from native code back to Flux source lines requires
   DWARF debug info in the LLVM emission. Not yet implemented.

---

## Rationale and alternatives

### Why Rust entry shim instead of C?

The runtime is written in Rust. A Rust entry shim has direct access to
`JitContext::new()`, the GC heap, and all runtime types. A C shim would
need a separate FFI boundary for initialization, adding complexity.

### Why not compile Flux to Rust source?

Compiling to Rust source (like Koka compiles to C) would leverage `rustc`
for optimization and linking. However:
- The Flux runtime uses `Rc`, `Value` enum, GC — these are Rust-native already
- LLVM gives better control over code generation and optimization
- The LLVM backend already exists and works

### Why not compile to C like Koka?

Koka generates readable C code and lets GCC/Clang optimize it. This is
effective but:
- C codegen is a new backend (LLVM backend already exists)
- Perceus reference counting (which makes Koka's C output efficient) is not
  implemented in Flux yet
- LLVM IR is lower-level and gives more direct control

### What if we don't do this?

Flux programs remain tied to the compiler binary. Distribution requires
shipping the full toolchain. Performance-critical deployments can't benefit
from AOT optimization.

## Prior art

| Language | AOT approach | Binary size |
|----------|-------------|-------------|
| **Koka** | Compile to C → GCC/Clang | ~1-2 MB (C runtime is small) |
| **GHC (Haskell)** | Native codegen or LLVM → link with RTS | ~5-10 MB |
| **OCaml** | Native compiler → link with runtime | ~1-3 MB |
| **Go** | Direct native codegen, static linking | ~3-5 MB |
| **Zig** | LLVM backend → static binary | ~1-2 MB |

Flux's approach (LLVM → .o → link with Rust runtime) is closest to GHC's
LLVM backend path. Binary sizes are comparable.

## Unresolved questions

1. **Runtime slimming** — can we tree-shake unused base functions to reduce
   binary size? If a program never uses `read_file`, don't link `io_ops.rs`.

2. **Function table format** — binary serialization format for JitFunctionEntry
   metadata. Should it be a simple flat array or a more structured format?

3. **Multi-module programs** — how does `flux build` handle `import` statements
   that reference other `.flx` files? Compile all to separate `.o` files and link?

4. **`rustc` dependency** — can we use `lld` or system `cc` instead of requiring
   the Rust toolchain for linking?

5. **Effect handler bootstrap** — standalone binaries need `IO` effect handling
   initialized before `main()` runs. How is this bootstrapped?

---

## Future possibilities

1. **Cross-compilation** — `flux build --target aarch64-unknown-linux-gnu` using
   LLVM's cross-compilation support. The LLVM backend already sets target triple.

2. **Incremental compilation** — cache `.o` files per module, only recompile
   changed modules.

3. **Dynamic libraries** — `flux build --shared program.flx -o libprogram.so` for
   embedding Flux code in Rust/C/Python projects.

4. **Perceus integration** — once proposal 0068-0070 land, the runtime shrinks
   significantly (no GC heap, no mark-and-sweep) and binary size drops.

5. **Debug info** — emit DWARF debug info mapping LLVM IR back to Flux source
   lines, enabling `gdb`/`lldb` debugging of standalone binaries.

6. **Package manager integration** — `flux install` compiles and installs a
   standalone binary to `~/.flux/bin/`.
